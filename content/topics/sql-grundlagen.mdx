SQL sieht aus wie eine Sprache, ist aber fünf. Jede Anweisung gehört zu genau einer
Teilsprache – und daraus folgt fast alles, was in der Prüfung dazu gefragt wird.

<Callout type="merksatz" title="Der eine Gedanke">
  Ordne jede Anweisung ihrer Teilsprache zu, dann weißt du sofort, was sie anfasst
  (**Struktur** oder **Daten**) und ob du sie **zurücknehmen** kannst. Genau daran hängt
  die häufigste Fangfrage: Ein `DELETE` lässt sich per `ROLLBACK` rückgängig machen, ein
  `DROP TABLE` nicht.
</Callout>

## Die fünf Teilsprachen

| Kürzel | Steht für | Anweisungen | Fasst an |
| --- | --- | --- | --- |
| **DDL** | Data **Definition** Language | `CREATE`, `ALTER`, `DROP`, `TRUNCATE` | die Struktur |
| **DML** | Data **Manipulation** Language | `INSERT`, `UPDATE`, `DELETE` | die Daten |
| **DQL** | Data **Query** Language | `SELECT` | liest nur |
| **DCL** | Data **Control** Language | `GRANT`, `REVOKE` | die Rechte |
| **TCL** | **Transaction** Control Language | `COMMIT`, `ROLLBACK`, `SAVEPOINT` | die Klammer drumherum |

<Callout type="tipp" title="Merkhilfe">
  **D**efinition – **M**anipulation – **Q**uery – **C**ontrol – **T**ransaction.
  Wer das Wort hinter dem Buchstaben kennt, kann jede Anweisung selbst zuordnen, statt
  Listen auswendig zu lernen. `SELECT` **fragt ab** → Query. `GRANT` **kontrolliert** den
  Zugriff → Control.
</Callout>

## Kernbegriffe

<TermGrid>
  <TermCard term="Constraint">
    Regel, die die Datenbank selbst durchsetzt: `NOT NULL`, `UNIQUE`, `PRIMARY KEY`,
    `FOREIGN KEY`, `CHECK`, `DEFAULT`. Verhindert falsche Daten, bevor sie entstehen.
  </TermCard>
  <TermCard term="Primärschlüssel">
    Identifiziert eine Zeile eindeutig. Implizit `NOT NULL` **und** `UNIQUE` – beides
    zusammen, nicht nur eins davon.
  </TermCard>
  <TermCard term="Referenzielle Integrität">
    Ein Fremdschlüssel darf nur auf einen existierenden Datensatz zeigen. Die Datenbank
    wacht darüber, nicht die Anwendung.
  </TermCard>
  <TermCard term="Transaktion">
    Mehrere Anweisungen als eine Einheit: entweder alle oder keine. Abgeschlossen mit
    `COMMIT`, verworfen mit `ROLLBACK`.
  </TermCard>
  <TermCard term="Implizites COMMIT">
    DDL-Anweisungen schließen die laufende Transaktion automatisch ab. Danach ist kein
    `ROLLBACK` mehr möglich.
  </TermCard>
  <TermCard term="DEFAULT">
    Wert, den die Datenbank einsetzt, wenn beim `INSERT` keiner angegeben wurde.
  </TermCard>
</TermGrid>

## Tabellen anlegen (DDL)

```sql
CREATE TABLE mitarbeiter (
  id        INTEGER PRIMARY KEY,
  name      TEXT    NOT NULL,
  email     TEXT    UNIQUE,
  gehalt    INTEGER CHECK (gehalt > 0),
  aktiv     INTEGER DEFAULT 1,
  abt_id    INTEGER REFERENCES abteilung(id)
);
```

Gängige Datentypen – die genauen Namen unterscheiden sich je nach Datenbanksystem, die
Kategorien nicht:

| Kategorie | Typisch | Wofür |
| --- | --- | --- |
| Ganzzahl | `INTEGER`, `INT`, `SMALLINT` | Schlüssel, Stückzahlen |
| Kommazahl | `DECIMAL(8,2)`, `NUMERIC` | **Geldbeträge** |
| Fließkomma | `FLOAT`, `REAL` | Messwerte |
| Text | `VARCHAR(50)`, `CHAR(5)`, `TEXT` | Namen, Beschreibungen |
| Datum/Zeit | `DATE`, `TIME`, `TIMESTAMP` | Termine |
| Wahrheitswert | `BOOLEAN` | Ja/Nein-Felder |

<Callout type="achtung" title="Geld nie als FLOAT">
  `FLOAT` und `REAL` rechnen binär und können `0.1 + 0.2` nicht exakt darstellen. Bei
  Beträgen führt das zu Rundungsfehlern, die sich aufsummieren. Richtig ist
  `DECIMAL(8,2)` – acht Stellen insgesamt, davon zwei nach dem Komma.

  `VARCHAR(50)` speichert nur die tatsächliche Länge, `CHAR(5)` füllt immer auf fünf
  Zeichen auf. `CHAR` also nur bei fester Länge, etwa einem Ländercode.
</Callout>

## Struktur ändern und löschen

```sql
ALTER TABLE mitarbeiter ADD COLUMN eintritt DATE;
DROP TABLE mitarbeiter;
```

| Anweisung | Wirkung | Umkehrbar? |
| --- | --- | --- |
| `DELETE FROM t` | löscht Zeilen, Tabelle bleibt | ja, per `ROLLBACK` |
| `TRUNCATE TABLE t` | leert die Tabelle schnell | **nein** (DDL) |
| `DROP TABLE t` | löscht Zeilen **und** Struktur | **nein** (DDL) |

Diese drei Zeilen sind eine der häufigsten Prüfungsfragen zum Thema überhaupt.

## Rechte vergeben (DCL)

```sql
GRANT SELECT, INSERT ON mitarbeiter TO azubi;
REVOKE INSERT ON mitarbeiter FROM azubi;
```

Der Gedanke dahinter ist das **Prinzip der geringsten Rechte**: Jede Rolle bekommt genau
die Rechte, die sie zum Arbeiten braucht – nicht mehr. Ein Auswertungskonto braucht
`SELECT`, kein `DELETE`.

## Transaktionen (TCL)

```sql
BEGIN;
  UPDATE konto SET stand = stand - 100 WHERE id = 1;
  UPDATE konto SET stand = stand + 100 WHERE id = 2;
COMMIT;   -- oder ROLLBACK, dann passiert nichts von beidem
```

Das klassische Beispiel: Bei einer Überweisung darf niemals nur die Abbuchung
ausgeführt werden. Entweder beide `UPDATE` oder keins.

<Callout type="achtung" title="DDL beendet die Transaktion">
  Ein `CREATE`, `ALTER`, `DROP` oder `TRUNCATE` löst in den meisten Datenbanksystemen ein
  **implizites COMMIT** aus. Alles davor ist dann festgeschrieben, und ein `ROLLBACK`
  danach holt nichts mehr zurück.

  Deshalb gilt in der Prüfung: „Kann man ein versehentliches `DROP TABLE` rückgängig
  machen?" – Nein. Nur eine Datensicherung hilft.
</Callout>

## Stolperfallen

<Callout type="achtung" title="Die sechs teuersten Fehler">
  1. **`DELETE` und `TRUNCATE` verwechselt.** `DELETE` ist DML und umkehrbar,
     `TRUNCATE` ist DDL und nicht.
  2. **Geldbeträge als `FLOAT`.** Rundungsfehler, die sich aufsummieren – nimm `DECIMAL`.
  3. **`PRIMARY KEY` zusätzlich als `NOT NULL UNIQUE` deklariert.** Doppelt gemoppelt;
     der Primärschlüssel ist beides schon.
  4. **Fremdschlüssel ohne passenden Datentyp.** `abt_id INTEGER` muss zu
     `abteilung.id INTEGER` passen, sonst scheitert die Verknüpfung.
  5. **`ROLLBACK` nach einer DDL-Anweisung erwartet.** Das implizite COMMIT hat da längst
     zugeschlagen.
  6. **`SELECT` der DML zugeordnet.** `SELECT` liest nur und ist DQL – es verändert nichts.
</Callout>
