Pseudocode ist die Brücke zwischen Gedanke und Programm: strukturiert genug, um einen
Algorithmus eindeutig zu beschreiben, aber ohne die Syntaxregeln einer konkreten Sprache.
In der Prüfung ist er der Standard, weil er unabhängig davon funktioniert, ob du Java,
C# oder Python gelernt hast.

<Callout type="merksatz" title="Warum dieses Thema so viel bringt">
  Tracen ist eine **mechanische Fertigkeit**, kein Ratespiel. Wer die Wertetabelle
  diszipliniert führt, löst auch Aufgaben zu Algorithmen, die er noch nie gesehen hat.
  Genau deshalb lohnt sich Üben hier mehr als bei jedem anderen Thema.
</Callout>

## Notation

Die IHK nutzt eine deutschsprachige Notation, die **nicht bis ins Detail normiert** ist.
Diese Varianten solltest du lesen und schreiben können:

| Konstrukt | Schreibweise |
| --- | --- |
| Zuweisung | `summe ← 0` (auch `:=` oder `=`) |
| Verzweigung | `WENN … DANN` / `SONST WENN` / `SONST` / `ENDE WENN` |
| Kopfgesteuerte Schleife | `SOLANGE … TUE` / `ENDE SOLANGE` |
| Fußgesteuerte Schleife | `WIEDERHOLE` / `BIS …` |
| Zählschleife | `FÜR i VON 1 BIS n [SCHRITT s] TUE` / `ENDE FÜR` |
| Funktion | `FUNKTION name(p)` / `GIB x ZURÜCK` / `ENDE FUNKTION` |
| Ausgabe | `GIB x AUS` |
| Operatoren | `MOD`, `DIV`, `UND`, `ODER`, `NICHT` |
| Vergleich | `=`, `<>`, `<`, `>`, `<=`, `>=` |

<Callout type="achtung" title="Kopf- oder fußgesteuert – der Unterschied zählt">
  `SOLANGE` prüft **vor** dem ersten Durchlauf und kann null Mal laufen.
  `WIEDERHOLE … BIS` prüft **danach** und läuft immer mindestens einmal.

  Klassische Prüfungsfrage: Was gibt `WIEDERHOLE … BIS i > 5` aus, wenn `i` schon bei 10
  startet? Antwort: einen Durchlauf. Probier es unten im Tracer aus.
</Callout>

## Der Schreibtischtest

Die Methode ist immer dieselbe:

1. Für jede Variable eine Spalte anlegen, dazu eine Spalte „Ausgabe".
2. Startwerte eintragen.
3. Zeile für Zeile abarbeiten und **nach jeder Zuweisung** eine neue Zeile schreiben.
4. Bei jeder Bedingung notieren, ob sie wahr oder falsch war.
5. Nichts im Kopf rechnen – genau da entstehen die Fehler.

Der Tracer unten macht exakt das automatisch. Nutze ihn zum **Kontrollieren**, nicht zum
Ersetzen: In der Prüfung führst du die Tabelle von Hand.

<PseudocodeTracer
  title="Beispiel: Summe von 1 bis 5"
  task="Geh Schritt für Schritt durch und beobachte, wie summe und i sich verändern. Die Spalte 'Aktion' zeigt, was jeder Schritt bewirkt hat."
  code={`summe ← 0
FÜR i VON 1 BIS 5 TUE
  summe ← summe + i
ENDE FÜR
GIB summe AUS`}
  variables={['i', 'summe']}
/>

## Schleifen richtig zählen

Wie oft läuft eine Schleife? Das ist die häufigste Teilfrage überhaupt:

| Kopf | Anzahl Durchläufe |
| --- | --- |
| `FÜR i VON 1 BIS n` | `n` |
| `FÜR i VON 0 BIS n` | `n + 1` |
| `FÜR i VON 0 BIS n - 1` | `n` |
| `FÜR i VON 1 BIS n SCHRITT 2` | `n / 2` aufgerundet |
| `FÜR i VON n BIS 1 SCHRITT -1` | `n` |

<Callout type="tipp" title="Wenn du unsicher bist">
  Setz die kleinste sinnvolle Zahl ein – meist `n = 1` oder `n = 2` – und zähl von Hand
  durch. Das dauert zehn Sekunden und ist zuverlässiger als jede Formel im Kopf.
</Callout>

## Funktionen

```
FUNKTION fakultaet(n)
  WENN n <= 1 DANN
    GIB 1 ZURÜCK
  ENDE WENN
  GIB n * fakultaet(n - 1) ZURÜCK
ENDE FUNKTION
```

Beim Tracen einer Funktion gilt: Jeder Aufruf hat seine **eigenen** lokalen Variablen. Ein
`n` im Aufruf `fakultaet(3)` hat nichts mit dem `n` in `fakultaet(2)` zu tun. In der
Wertetabelle unten erkennst du die Aufruftiefe an den vorangestellten Zeichen `›`.

<PseudocodeTracer
  title="Rekursion beobachten"
  task="Verfolge, wie sich die Aufrufe stapeln - und wie die Rückgabewerte danach von innen nach außen zurücklaufen."
  code={`FUNKTION fakultaet(n)
  WENN n <= 1 DANN
    GIB 1 ZURÜCK
  ENDE WENN
  GIB n * fakultaet(n - 1) ZURÜCK
ENDE FUNKTION

GIB fakultaet(4) AUS`}
/>

## Stolperfallen

<Callout type="achtung" title="Die sechs teuersten Fehler">
  1. **Off-by-One.** `VON 1 BIS n` läuft `n` Mal, `VON 0 BIS n` läuft `n + 1` Mal. Bei
     Arrays fast immer die Fehlerquelle Nummer eins.
  2. **Endlosschleife.** Die Bedingung wird im `SOLANGE`-Rumpf nie verändert – `i ← i + 1`
     vergessen.
  3. **Indexgrenzen.** Bei einem Array der Länge `n` mit Start bei 0 ist der letzte
     gültige Index `n - 1`. Steht in der Angabe nichts, schreib deine Annahme dazu.
  4. **Zuweisung statt Vergleich.** `←` weist zu, `=` vergleicht. In Bedingungen nie `←`.
  5. **Fehlender Abschluss.** Jedes `WENN` braucht `ENDE WENN`, jede Schleife ihr Ende.
     Sauber einrücken – das findet den Fehler von selbst.
  6. **Variable nicht initialisiert.** `summe ← summe + i`, ohne dass `summe` vorher einen
     Wert bekam.
</Callout>

<FreeText
  question={
    <>
      Erkläre in zwei bis drei Sätzen, warum ein Algorithmus <strong>terminieren</strong>{' '}
      muss und woran du im Pseudocode erkennst, dass eine Schleife das nicht tut.
    </>
  }
  solution={
    <>
      Ein Algorithmus muss nach endlich vielen Schritten enden, sonst liefert er nie ein
      Ergebnis und erfüllt die Definition nicht. Im Pseudocode erkennt man drohende
      Nicht-Terminierung daran, dass keine Anweisung im Schleifenrumpf die Abbruchbedingung
      beeinflusst – typischerweise, weil die Zählvariable nicht verändert wird oder in die
      falsche Richtung läuft.
    </>
  }
/>

<FreeText
  question={
    <>
      In Übung 4 lieferte der Algorithmus 9 statt 15. Benenne <strong>beide</strong>
      {' '}Schwachstellen des Codes – die, die den Fehler verursacht, und die, die bei
      anderen Daten zum Problem würde.
    </>

  solution={
    <>
      <strong>Der eigentliche Fehler:</strong> Die Schleife läuft nur{' '}
      <code>VON 0 BIS 1</code> und betrachtet damit nur die Indizes 0 und 1. Das größte
      Element steht aber an Index 2 und wird nie angeschaut – ein klassischer
      Off-by-One-Fehler durch eine fest eingetippte Obergrenze. Richtig ist{' '}
